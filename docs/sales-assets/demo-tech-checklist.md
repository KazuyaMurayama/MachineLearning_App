## 3. デモ中の操作順序（設計方針）

デモは「重い処理を先に走らせ、その間に軽い画面で語る」という原則で設計しています。これは、初回レンダリングで遅延しがちなクエリを早めに発火させ、説明中にキャッシュを温める狙いがあります。

推奨順序:

1. **冒頭の挨拶・アジェンダ説明（2分）** — この間に裏でダッシュボードを開いておく。
2. **事務所/経営ダッシュボード表示（3分）** — 最初の重いクエリ。読み込み中は「初回集計のため数秒お待ちください」とナレーション。
3. **AIブリーフィングのデモ（3分）** — テキスト生成中に裏で次画面をプリロード。
4. **LTV予測（士業）またはWhat-Ifシミュレーター（EC）（4分）**
5. **まとめとQ&A（3分）**

### EC向けWhat-Ifシミュレーターの特別運用

`ec-what-if` の予測モデルは現在 R²=-0.16 と精度再検証中です。本アプリを見せる際は必ず以下を守ってください。

- 数値予測を確定値として読み上げない。
- 「モデル再検証中です」「パイロット期間中に精度を確認します」と明確にナレーションする。
- 顧客から精度を問われた際は、「検証中」「パイロット期間で貴社データを用いて再評価します」と答える。
- スクリーンショットで結果を残して顧客に渡さない。

## 4. トラブル時のフォールバック手順

### 4.1 画面が表示されない・著しく遅い

1. 画面を切り替えず、15秒待機する（ナレーションで場をつなぐ）。
2. 15秒経過しても改善しなければ、用意済みのスクリーンショットに切り替える。
3. 「こちらは事前に取得した画面です」と正直に伝える。
4. デモ後、ログを確認して原因を `#demo-issues` に報告する。

### 4.2 ネットワーク障害

1. モバイルルータに切り替える。
2. 切替中は会社紹介スライドで時間を稼ぐ。
3. 復旧しない場合はローカルキャッシュ版デモに切り替える。

### 4.3 予測モデルがエラーを返す

1. 画面遷移せず、「ただいまモデルの再検証中のため、時折このような表示が出ます」と伝える。
2. 他のアプリ（ダッシュボードまたはAIブリーフィング）に話題を誘導する。
3. エラーの再現条件を記録し、開発チームに連携する。

### 4.4 全体フォールバック

1. 事前に用意したスライド版デモ資料に切り替える。
2. 顧客に後日ライブデモの再実施を提案する。

## 5. 既知の問題と迂回策

| 問題 | 影響 | 迂回策 |
| --- | --- | --- |
| ダッシュボード初回ロードが遅い（5〜15秒） | 最初のインパクトが弱まる | 1時間前にウォームアップ実施。冒頭ナレーションで時間を稼ぐ |
| `ec-what-if` モデルの R²=-0.16 | 予測値の信頼性欠如 | 「精度再検証中」と必ずナレーション、数値を確定値として提示しない |
| AIブリーフィングのテキスト生成に揺らぎ | 毎回違う出力 | 事前に良い出力例を1つキャプチャしておきフォールバック素材に |
| ブラウザ拡張による表示崩れ | レイアウト崩壊 | デモ用プロファイルを使う（拡張なし） |

## 6. 使用禁止ワード

デモ中および質疑応答中、以下の表現は使用禁止です。過剰な約束は後続トラブルの原因になります。

- 「100%」
- 「絶対」
- 「保証します」
- 「確実に改善します」
- 「必ず黒字化します」
- 「どんなデータでも対応可能です」

代替表現:

- 「過去の検証事例では〜」
- 「貴社のデータで再計算する前提で〜」
- 「現時点の仮説としては〜」
- 「パイロット期間で検証したいポイントです」
