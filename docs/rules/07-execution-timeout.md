# 07: 実行計画・タイムアウト対策ルール

## タスク分割

- タスクを **細かいサブタスク** に分割し、各ステップで **保存・コミット** しながら進める。
- 長時間処理が見込まれる場合は、**チェックポイントを明示的に計画** してから実行する。
- 1 タスク完了 → commit → 次タスク の繰り返しで途中停止リスクを最小化。

## API Stream idle timeout 対策

| 原因 | 対策 | 検証方法 |
|------|------|---------|
| MCP 切断 | 書き込み前に **軽量 read でプロービング** | SHA 取得成功を確認 |
| 大ファイル push | 1 ファイル **≤ 4KB 目標、最大 8KB** | ファイルサイズを事前計算 |
| 複数ファイル並列 | `create_or_update_file` で **1 ファイルずつ** 直列処理 | 各コミット SHA を確認 |
| 書き込み確認 | push 後に **read verify** で SHA 変化を確認 | 旧 SHA ≠ 新 SHA |
| 失敗時の即再送 | 失敗 → **待機 → 再 probe → 再送** の段階的対応 | probe 成功後に再送 |

## push_files 使用時の上限

- 同時に含めるファイル数: **最大 3 ファイル**
- 合計サイズ: **≤ 10KB**
- 超える場合は複数回に分割

## タイムアウト発生時のフォールバック

1. まず MCP プロービング（軽量 read）を試行
2. 成功なら送信サイズを半分以下にして再試行
3. 3 回連続失敗ならチャット内にファイル内容を提示し、ユーザーに手動保存を依頼

## セッション中断・リカバリプロトコル

### 自動化の限界（重要）

- タイムアウトはシステムが強制終了するため、**Claude 自身がリアルタイムに検出することは不可**
- 代替手段: **`progress.md` への定期書き込み + git commit チェックポイント**
- ユーザー操作 `claude --resume` で直前セッションのコンテキストも復元可能

### 実行中の義務（長時間タスク）

- **フェーズ完了ごと** に `progress.md` を更新してコミットする
- `progress.md` に記録する内容:
  - 完了済みフェーズ（`[x]`）と commit SHA
  - 次に実行すべき手順（`→ 次:`）
  - 再開に必要な状態メモ（変数値・ファイルパス等）

### セッション起動時（CLAUDE.md の起動チェックリスト参照）

1. `progress.md` を読む
2. 「進行中タスクなし」以外の内容があれば **続きから再開する**
3. タスク完了後は `progress.md` を「進行中タスクなし」にリセットしてコミット

### `claude --resume` との組み合わせ

```
ユーザーが claude --resume を実行
  → 直前セッションのコンテキスト復元
  → Claude が progress.md を読む
  → 再開点を即座に特定して続行
```

この組み合わせが最も確実なリカバリ手順。
