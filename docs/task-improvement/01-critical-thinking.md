## エグゼクティブサマリー

3タスクはいずれも「完了」と報告されているが、実際にコードを動かして精査すると、品質スコア・L3到達宣言・動作確認のいずれも**重大な過信**が含まれる。最も深刻なのはモデル品質の問題だ。ec-what-if の離脱率モデルは R² = -0.16（ランダム予測以下）であり、これを「L3キラー機能・94/100点」と称することは顧客への虚偽説明になりかねない。「実装した」と「動く」と「使える」は三段階の別物であり、現段階の完了報告はその最初の段階に過ぎない。パイロット顧客ゼロの状態で「L3実装完了」「月額26〜30万円」を市場に訴求することは、信頼毀損リスクを孕む。

## TASK 1: デプロイ前動作確認への批判

### 発見した問題点

- 🔴 **[重大] ast.parse() を「動作確認」と呼ぶのは完全な誤称である**
  - 構文チェックは「Pythonファイルとして読める」ことを確認するだけだ。`import lightgbm` が失敗する環境、`sample_data/training_data.csv` が存在しない状態での実行、Streamlit固有の再実行フロー（`@st.cache_resource` の挙動）は一切検証されていない。
  - 実証: `ec-what-if/app.py` は起動時に `data_path = os.path.join(...)` で CSVの存在チェックを行い、なければ `st.stop()` で停止する。これは ast.parse() では検出できない。

- 🔴 **[重大] ec-ad-roi と ec-rfm に `create_sample_data.py` が存在するにも関わらず、TASK 1のチェックリスト「L3 6アプリの create_sample_data.py 実行確認」との整合が不明確**
  - 実際にディレクトリを確認すると `ec-ad-roi/` と `ec-rfm/` は `create_sample_data.py` を持っている。しかし「L3 6アプリ」という表現が「新規実装のL3アプリだけ」を指すのか「全create_sample_data.py保有アプリ」を指すのか曖昧であり、対象が明記されていない。

- 🔴 **[重大] 離脱率モデルの R² が -0.16 という致命的な数値が「動作確認済み」として見落とされた**
  - 実際にモデルを学習させると: 売上 R²=0.45、ROAS R²=0.14、**離脱率 R²=-0.16**（ランダム予測以下）。
  - 離脱率データに「1.0を超える値が 27件（8.3%）」存在する。離脱率は定義上0〜1の確率値であり、1.482という値は物理的に意味をなさない異常値だ。
  - ast.parse() はこの問題を検出できない。「サンプルデータが想定通りの分布か」を誰も確認していない。

- 🟡 **[中程度] import健全性確認の「健全性」の定義が不明**
  - `import lightgbm` `import shap` が通るかを確認したとしても、バージョン互換性（LightGBM 4.x の API 変更、SHAP 0.42+でのTreeExplainer仕様変更）は検証されていない。

- 🟢 **[軽微] packages.txt の存在確認はしているが内容検証はゼロ**
  - `packages.txt` に書かれた apt パッケージ（フォント等）がStreamlit Cloud の Ubuntu バージョンで利用可能かは未確認。

### 見落としリスク

- **Streamlit Cloud 特有の制約**: `@st.cache_resource` はサーバ再起動で無効化される。本番で初回学習に30〜60秒かかる場合、顧客はスピナーを見て「壊れた」と判断して離脱する。
- **日本語フォント問題**: `setup_japanese_font()` が `shigyou-ltv` と `ec-what-if` の両方で実装されているが、Streamlit Cloud の標準イメージには Noto Sans CJK JP が含まれていない可能性が高い。フォントフォールバックで英語表示になりグラフタイトルが文字化けするリスクがある。
- **LightGBMの学習時間**: 150〜325行程度のデータでも、SHAP計算（TreeExplainer）は初回に数十秒かかることがある。`st.spinner` の表示はあるが、Streamlit Cloud の無料枠でCPUが遅い場合にタイムアウトするリスクが未評価だ。

### 推奨改善策

- `streamlit run app.py --server.headless true` をローカルまたはCIで実行し、起動エラーの有無を確認する
- サンプルデータの統計的妥当性チェックを `create_sample_data.py` に組み込む（離脱率は `clip(0, 1)` など）
- ec-what-if の離脱率モデルについて、`R² < 0` の場合に警告バナーを表示するコードを追加する

## TASK 2: 17アプリ統合動作確認への批判

### 発見した問題点

- 🔴 **[重大] 「ec-demo の欠落を発見した」という事実は、チェック手法そのものの信頼性を崩している**
  - 1アプリが漏れていたなら、残り17（→18）アプリの確認も同じ手法で行ったはずであり、「漏れゼロ」という保証はない。漏れが発見されたことは「チェックプロセスが確実ではなかった」ことの証拠だ。修正コミット `99fca76` は対症療法であり、根本原因（手動チェックの限界）は未解決である。

- 🔴 **[重大] URLが実際に到達可能かを検証していない**
  - APP_URLS / EC_URLS に定義されたURLはいずれも `https://<subdomain>.streamlit.app` 形式だが、これらのサブドメインが Streamlit Cloud 上で実際に予約・デプロイ済みかを確認する手段が取られていない。
  - 現時点でポータルのリンクをクリックすると「Not Found」になる可能性が100%残っている。

- 🔴 **[重大] EC ポータルのアプリ数表示に矛盾がある**
  - `ec_app.py` のdocstring・サイドバー・KPIカードはいずれも「6アプリ」と表示しているが、`EC_URLS` には `ec-demo` を含む **7エントリ**が存在し、ページ本文にもec-demoのカードが表示される。
  - TASK 2 で ec-demo を「追加した」のに、カウント表示が更新されていない。これは TASK 2 の「整合性チェック」が完了していないことを示す。

- 🟡 **[中程度] ディレクトリ名とURLサブドメイン名が 10/18 で不一致**
  - `shigyou-demo` → `shigyou-churn`、`report-gen` → `shigyou-report`、`ec-rfm` → `ec-rfm-analysis` など、ディレクトリ名とデプロイ先URLが一致しない組み合わせが多数存在する。
  - Streamlit Cloud のデプロイ設定でmain fileパスを誤って設定するリスクや、URLを人に伝える際に混乱が生じるリスクがある。

- 🟡 **[中程度] `streamlit_app.py` のdocstringが「9アプリの専用ポータル」と記載されているが実際は11アプリ**
  - ファイル先頭のdocstring: `"税理士・社労士・行政書士向けAIツール9アプリの専用ポータル"` → KPIカードは「11」。
  - コードレビューをした際に混乱の原因になる典型的な「嘘をつくコメント」だ。

### 見落としリスク

- **ポータル自体のデプロイ**: `streamlit_app.py` と `ec_app.py` がそれぞれどのURLでデプロイされるかが定義されていない。アプリカードに個別URLを貼ったとしても、そのポータルへの入口URLが顧客に渡らなければ意味がない。
- **リンク切れの発覚タイミング**: パイロット顧客に見せる段階で初めてURLが404だと判明した場合、信頼を一気に失う。「事前確認済み」という安心感が逆に確認を怠らせるリスクがある。

### 推奨改善策

- `curl -o /dev/null -s -w "%{http_code}" <URL>` などで全URLの到達確認を自動化するスクリプトを作成する
- EC ポータルの「6アプリ」表示を「7アプリ」に修正し、docstringとKPIカードを一致させる
- `streamlit_app.py` のdocstringを「9アプリ」→「11アプリ」に修正する
- チェックリストをコードで管理する（ディレクトリ名とEC_URLS/APP_URLSのキーが一致しているかをテストとして実行する）

## TASK 3: ビジネス資料更新への批判

### 発見した問題点

- 🔴 **[重大] 「L3実装完了」という表現は事実を誤解させる**
  - `shigyou-business-alignment.md` セクション5の現状評価に「L3は✅実装完了（3本）」と記載されているが、同文書のセクション4.3に「Month 5-6: L3実証 — L3パイロット顧客1社での運用」とある。
  - 「実装完了」と「L3サービス達成」は別概念だ。ツールを作っただけでL3（AI経営パートナー月額30万円）に「到達」したとは言えない。ソフトウェアを開発することと、顧客がそれに月30万円を支払うことは全く異なる。
  - 「完了」という言葉が次フェーズの危機感を薄める。パイロット顧客が0社の時点での「到達宣言」は楽観バイアスの典型だ。

- 🔴 **[重大] ec-business-alignment.md の「急務3項目が解消済み」という記述が不正確**
  - 急務①: 「逆SHAP What-Ifシミュレーター実装 → ec-what-if で完成」と記載されているが、実際に学習したモデルの精度（R² = -0.16 for 離脱率）を考えると、「完成」とは言えない。R²が負のモデルによるWhat-If提案は、ランダムよりも悪いアドバイスを与える可能性がある。
  - 急務②: 「共通データ基盤構築」として ec-executive-dashboard が解消とされているが、同文書のセクション4「改善計画」には「全アプリの共通データ基盤構築」が優先度4（難易度S=1週間超）で未完と記載されている。「1画面に4アプリデータを統合表示する」ことと「横断分析を可能にする共通データ基盤」は別物であり、ダッシュボードがデータを静的に表示するだけでは横断分析は実現していない。

- 🟡 **[中程度] 品質スコア「91-95/100」の根拠が存在しない**
  - 誰が、何の基準で採点したか不明だ。自己採点であれば楽観バイアスが最大になる。
  - 「ビジネス訴求力」「LightGBM実装の完成度」「SHAP解釈の品質」を客観基準で採点するには、少なくとも複数名のレビュー、または定量的なテスト（精度指標、UIテスト結果）が必要だ。
  - R² = -0.16 のモデルを含むアプリが「94/100」を得るのは採点基準として破綻している。

- 🟡 **[中程度] 月額価格（30万円 / 26万円）は根拠なき仮説である**
  - 文書内では「競合比較」として正当化されているが、実際の顧客に提示したことは一度もない。価格は市場が決める。「データサイエンティスト採用より安い」という比較軸は、中小士業事務所が「データサイエンティスト採用」を真剣に検討しているという前提の上に成り立つが、その前提自体が検証されていない。

- 🟢 **[軽微] 文書の更新日と本文の整合性の部分的ズレ**
  - `ec-business-alignment.md` のフッター: `"2026-04-08にL3実装完了を反映"` は正しい。しかし冒頭の `"作成日: 2026-04-06"` と組み合わせると、どの内容が元来の分析でどこが追記なのかが後から追跡しにくい。変更履歴として section を分けるべきだった。

### 見落としリスク

- **残課題の矮小化**: `ec-business-alignment.md` の結論部に「ec-dashboard へのML予測+SHAP追加は未着手 — 将来対応」と1行で片付けられているが、ec-dashboard は L1 スターターパックのコア機能に相当し、充足率 45% と最も低い。これが未対応のまま L3 完了を謳うのは、土台が崩れたままビルを立てるに等しい。
- **improvement-backlog.md の残項目**: バックログには RFM / ec-dashboard / ec-ad-roi などに対して🔴（最優先）の未実装項目が多数残っている。これらを「L3完了後の残課題」として整理しているが、多くはL1/L2の基盤機能であり、L3より先に対処すべき項目だ。

### 推奨改善策

- 「L3実装完了」という表現を「L3機能プロトタイプ作成完了（パイロット顧客検証前）」に改める
- 品質スコアの採点基準を明文化し、モデル精度（R², MAE）を品質スコアの必須要素とする
- `ec-dashboard` の ML予測追加を「L3完了後の将来対応」から「L1スターター完成の必須要件」に再分類する

## 最も致命的な見落とし TOP3

### 1. 離脱率モデルがランダム予測以下（R² = -0.16）であることを誰も検証せずに「L3キラー機能」と呼んでいる

ec-what-if のWhat-Ifシミュレーターはec側のL3の目玉として「GA4・Lookerにない唯一無二の機能」と位置づけられている。しかし実際に学習を走らせると、離脱率モデルの R² は -0.16 だ。これはランダムに数字を出すよりも悪い予測精度を意味する。さらにトレーニングデータの離脱率に1.0を超える値（最大1.482）が27件存在しており、データ生成ロジック自体に欠陥がある。顧客に「広告費を変えると離脱率がこう変わります」と見せるシミュレーターが、実態としてでたらめな予測を返す可能性がある。ast.parse() 動作確認ではこれを検出できない。この問題はデプロイ後に「デモをやったら変な結果が出た」という形で顕在化する。

### 2. Streamlit Cloud のURL（サブドメイン）が実際に確保されているかを誰も確認していない

18本のアプリに対してURLが定義されているが、これらは「定義されたURL」であり「実際にアクセス可能なURL」ではない。Streamlit Cloud では GitHub リポジトリをデプロイして初めてサブドメインが有効になる。`shigyou-churn.streamlit.app` や `ec-rfm-analysis.streamlit.app` 等の全URLについて、現時点でのデプロイ状況が確認されていない。パイロット顧客に「これを見てください」と伝えたURLが全てNot Foundだった場合、信頼は即座に失われる。ec-demo の漏れ発見という事実が示すように、手動チェックには系統的な欠落が生じる。

### 3. 「L3実装完了」という宣言がパイロット顧客獲得への危機感を消失させる

`shigyou-business-alignment.md` は「L3到達の核心課題（解消済み）」「実装完了」と繰り返し宣言している。これは次のフェーズ（パイロット顧客獲得）に向けた動機づけを内部的に弱体化させる。実際のL3（月30万円・月26万円）は「ツールが動く」ことではなく「顧客が支払い続ける」ことで初めて成立する。shigyou の L3 ロードマップ（Month 5-6: L3パイロット1社）はまだ到達していないにも関わらず、文書上で「到達」と宣言することで「次にやるべきこと」の優先度が曖昧になり、最も重要なパイロット顧客獲得活動（実際の税理士事務所への訴求）が後回しになるリスクがある。

## 総合コメント

### 楽観バイアスの構造的問題

今回の3タスクには「実装した → 動く → 使える → 売れる」という4段階を混同する楽観バイアスが一貫して流れている。TASK 1 の「全18アプリの動作確認」は構文チェックにとどまり、TASK 3 の「L3実装完了」はプロトタイプ作成にとどまる。品質スコア「91-95/100」という数字は、採点者が実装者本人（あるいは実装を知るエージェント）である限り、参照値として意味をなさない。

### 次フェーズへの警告

次の行動として最も優先すべきは以下の3点であり、新機能追加よりも前に着手すべきだ:

1. **ec-what-if の離脱率データを修正し（clip(0,1)）、モデルを再学習させ、R² ≥ 0.3 を達成してから「キラー機能」と呼ぶ**
2. **全18URLについて `curl` または Streamlit Cloud 管理画面でデプロイ状態を確認する。未デプロイのURLはポータルから一時的に非表示にする**
3. **ビジネス資料の「L3実装完了」を「L3プロトタイプ完成・パイロット検証中」に書き換え、実際の顧客反応を計測するプロセスを定義する**

実装の速度を落とす必要はないが、「完了」という言葉を使う基準を引き上げることが、次フェーズで信頼を失わないための最低限の防衛線だ。
