## エグゼクティブサマリー

5エージェントの評価が一致して示す最大の問題は、**「実装した」「動く」「使える」「売れる」の4段階を混同した楽観バイアス**の存在だ。R² = -0.16 の離脱率モデルを「L3キラー機能」と称し、構文チェック（ast.parse）を「動作確認完了」と呼び、パイロット顧客0社の状態で「L3実装完了・月額30万円」を宣言している。技術・品質・論理・ビジネス・UXのすべての観点から同じ構造的誤りが観測される。最優先で対処すべきは「ec-what-if の離脱率モデル欠陥」「Streamlit Cloud実デプロイ未確認」「Go-to-Marketアセットの完全不在」の3点であり、新機能追加より前に着手すべきである。

## 最優先問題（3人以上が指摘）

### 🔴 P0-1: ast.parse() を「動作確認」と呼ぶ誤称

- 指摘エージェント: Critical, Logical, Tech
- 核心: 構文チェックはランタイムエラー・依存ライブラリ問題・データ欠損・Streamlit Cloud固有の制約を一切検出できない。「デプロイ準備完了」の根拠にならない。
- 根拠:
  - Critical: "ast.parse() を『動作確認』と呼ぶのは完全な誤称。CSVの存在チェック、import失敗、Streamlit固有のフローは一切検証されていない"
  - Logical: "構文OK → 動作OK への推論は妥当でない。ランタイムエラーは構文チェックでは検出不能"
  - Tech: "ast.parse() による動作確認は構文レベルにすぎない。スモークテスト完全不在・CI/CDなし。4057行のプロダクションコードにテストゼロ"

### 🔴 P0-2: 離脱率モデル R² = -0.16 の見落とし

- 指摘エージェント: Critical, Logical, Tech
- 核心: ec-what-if のWhat-Ifシミュレーターを「L3キラー機能」と位置づけているが、離脱率モデルはランダム予測以下。サンプルデータに1.0超の異常値が27件存在し、データ生成ロジック自体に欠陥がある。
- 根拠:
  - Critical: "離脱率 R²=-0.16（ランダム予測以下）。離脱率データに1.0を超える値が27件（8.3%）存在。ast.parse()はこの問題を検出できない"
  - Logical: "採点基準なき品質スコア91-95において、R²=-0.16のモデルを含むアプリが94点を得るのは採点基準として破綻している"
  - Tech: "R² < 0の場合にst.warning()で警告バナーを表示する品質ゲートが必要。現在は無警告で顧客に表示される"

### 🔴 P0-3: 「L3実装完了」宣言の論理的破綻

- 指摘エージェント: Critical, Logical, Business, UX
- 核心: パイロット顧客0社・売上0円の状態で「L3実装完了・月額30万円」を宣言している。L3の本来の定義（月次レビュー・コンサル価値・実顧客運用）は一切充足されていない。
- 根拠:
  - Critical: "『実装完了』と『L3サービス達成』は別概念。ソフトウェアを開発することと、顧客がそれに月30万円を支払うことは全く異なる"
  - Logical: "『L3候補機能3本のプロトタイプを実装した』までは正当化可能。しかし『L3水準到達』という主張は反証不能な誇大表現"
  - Business: "技術側の文書はコミットハッシュを誇示するが、営業活動の記録は1行も存在しない。『売れる状態』と『売れている状態』を混同"
  - UX: "初見ユーザーが月額30万円のL3バナーを見て納得するまでの心理フローが一切未確認"

## 重要問題（2人が指摘）

### 🟡 P1-1: Streamlit Cloud 実デプロイ・URL疎通の未確認

- 指摘エージェント: Critical, Tech
- 核心: 18本のアプリに対してURLが定義されているが「定義されたURL」であり「実際にアクセス可能なURL」ではない。パイロット顧客にURLを渡した時点で全件「Not Found」になるリスクが残っている。

### 🟡 P1-2: 品質スコア91-95/100の根拠欠如

- 指摘エージェント: Critical, Logical
- 核心: 採点基準・採点者・採点プロセスが非開示。第三者が同一スコアを再現できず、検証不能な主張。対外的説得力ゼロ。既存アプリの充足率（45〜70%）と新規アプリのスコア（91〜95）の著しい差は基準の不統一を示唆する。

### 🟡 P1-3: ec-demo欠落が示すチェックプロセスの構造的信頼性問題

- 指摘エージェント: Critical, Tech
- 核心: 1アプリが漏れていたなら同一手法で確認した残り17アプリも「漏れゼロ」の保証がない。再発防止策（アプリ追加時の登録フロー）が定義されていない。

### 🟡 P1-4: Go-to-Marketアセットの完全不在

- 指摘エージェント: Business, UX
- 核心: 営業スライド・デモスクリプト・パイロット提案書・顧客向け料金表・ROI試算シートが全て存在しない。明日パイロット候補から「資料を送ってほしい」と連絡が来ても送れるものが何もない状態。

### 🟡 P1-5: 価格設定の論理的脆弱性

- 指摘エージェント: Logical, Business
- 核心: EC事業者の想定支払能力「月額8〜20万円」がL3価格「月26万円」を下回る内部矛盾。士業30万円の根拠「DS採用より安い」は比較対象として的外れ。いずれも市場検証ゼロ。

### 🟡 P1-6: DRY原則違反と保守性の問題

- 指摘エージェント: Tech, Logical
- 核心: `setup_japanese_font()` が6アプリに完全コピペ。EC系3アプリの共通マスター生成ブロックも複製。seed=42のランダム呼び出し順序がファイル間でずれ「同一データ」と主張しても実際の値が異なる可能性がある。

## 単独指摘（1人だが重要なもの）

### 🟢 UX-1: ec-what-if コールドスタートのデモ致命傷リスク（UX）
スライダー操作→リアルタイム予測というデモ最大の山場が、コールドスタートの沈黙で台無しになるリスク。段階的プログレス表示（`st.progress` + `st.status`）の追加コストは低い。

### 🟢 UX-2: briefing系2本のデモ映えの低さ（UX）
`shigyou-briefing`・`ec-monthly-briefing` はテキスト出力中心で視覚的インパクトがゼロ。月額26〜30万円のサービスを支えるには「おっ」ポイントが必要。

### 🟢 Tech-1: shigyou-ltv のキャッシュ設計の根本的誤り（Tech）
`st.session_state` にモデルを格納しているためセッション間でモデルが共有されず、ユーザーごとに毎回学習が走る。`@st.cache_resource` への移行が必要。

### 🟢 Tech-2: `subprocess.run()` の `check=True` 省略（Tech）
`ec-monthly-briefing` でスクリプト失敗時に生のPythonトレースバックがユーザーに表示される。4アプリで実装に一貫性がない。

## エージェント間の意見対立

### 対立点1: 「L3実装完了」は技術的に妥当か

- Tech Review の立場: 実装コードは存在する。品質スコア記録・コミットハッシュ・ファイル整合性は確認済み。技術タスクとしての「実装」は完了と評価できる。ただし本番品質ではない。
- Business Strategy の立場: 顧客0社・売上0円の状態での「完了」宣言は、スタートアップが陥る典型的な「作り続けることで前進している錯覚」。技術完成は「販売可能状態」であり「価値提供の完了」ではない。
- Logical Analyst の判定: 「L3候補機能3本のプロトタイプを実装した」という主張は根拠で正当化可能。しかし「L3水準到達」「L3実装完了」という表現は主張の範囲が証拠を超えており論理的飛躍がある。
- **統合判断**: TechレビューのOK判定は「コードが存在する」という限定的な意味では正しい。しかしBusinessとLogicalの指摘が妥当であり、「L3プロトタイプ実装完了（パイロット検証前）」に表現を修正すべき。現表現は次フェーズへの危機感を消失させる副作用があり、変更コストが低い割にリスクが高い。

### 対立点2: ec-what-if の「唯一無二」差別化主張は妥当か

- UX の立場: デモの最大山場として「スライダー操作→リアルタイム3モデル予測」は視覚的インパクトが最高。デモシナリオの中核に配置すべき。
- Business Strategy の立場: 「競合に機能がない」は「顧客がその機能に価値を感じる」を意味しない。DataRobotやSalesforce Einsteinが類似機能を提供している可能性を排除できない。EC事業者が「逆SHAP」を理解して購買決定するシナリオが未検証。
- **統合判断**: 技術的差別化の訴求はUXが正しい。ただし前提として離脱率モデル（R²=-0.16）の修正が必須。欠陥モデルのままでは「唯一無二」が「唯一無二のでたらめな予測」になる。BusinessのリスクはP0問題解決後に評価すべき二次的課題。

## 5エージェント全員の盲点

- **CI/CDの不在**: 5エージェントのうちTechが指摘したが、他4エージェントはCIパイプラインの欠如を独立した問題として取り上げていない。テストゼロ・CI ゼロの状態は月額30万円サービスの基盤として致命的なリスク。
- **Streamlit Cloud への実デプロイ検証**: 「URLが定義されている」と「実際にアクセス可能」の差を誰も実証的に確認していない。本番環境（Streamlit Cloud）での日本語フォント、メモリ上限、コールドスタート時間の実測値が存在しない。
- **顧客獲得プロセスの具体化**: BusinessはアウトリーチメールやデモシナリオのTODOを挙げているが、「誰が・どのルートで・どの見込み客に」当たるかの具体的なプロセスが5エージェントとも言及していない。コンタクトリスト、スコアリング基準、商談管理ツールの不在。
- **法的観点**: 顧客向け契約書、個人情報保護方針、SLA（稼働保証）の定義がどのエージェントも扱っていない。月額30万円のサービス契約に必要な法的基盤が未整備。

## 改善テーマのクラスタリング

### テーマA: 動作検証の信頼性
- 関連問題: P0-1, P1-1, P1-3, Tech-1, Tech-2
- 根本原因: 「構文OK = 動作OK」という誤った等式への依存。スモークテスト・CIパイプラインの不在。
- 対策方向: `streamlit run app.py --server.headless true` の全アプリ実行、CI導入、URL疎通確認スクリプトの自動化。

### テーマB: データ品質・サンプルデータの欠陥
- 関連問題: P0-2, DRY違反（seed不一致）
- 根本原因: `create_sample_data.py` の出力スキーマ・値域を誰も検証していない。離脱率に1.482が含まれることに誰も気づかなかった。
- 対策方向: `create_sample_data.py` にスキーマ検証（値域clip、型チェック）を組み込む。EC共通マスター生成を単一スクリプトに統合。

### テーマC: Go-to-Market アセット不在
- 関連問題: P1-4, P1-5
- 根本原因: 技術タスクにリソースを集中し、営業準備活動をゼロに近い状態に放置している構造的配分ミス。
- 対策方向: パイロット提案書・ROI試算シート・15分デモスクリプト・料金表を1週間以内に作成。技術実装を一時停止して営業活動にシフト。

### テーマD: コード重複と保守性
- 関連問題: P1-6（DRY違反）, Tech-2（エラーハンドリング不統一）
- 根本原因: 共通ライブラリ設計なしに各アプリを独立実装したモジュール設計の欠如。
- 対策方向: `apps/common/utils.py` に `setup_japanese_font()` を集約。EC3アプリの共通マスター生成を単一スクリプトに統合。エラーハンドリングを `shigyou-office-dashboard` パターンに統一。

### テーマE: UX実測不足
- 関連問題: UX-1（コールドスタート）, UX-2（briefing系のデモ映え）
- 根本原因: 実ブラウザでのユーザー体験を一切確認せずに「動作確認完了」としている。特にデモ文脈での感情の流れが設計されていない。
- 対策方向: ec-what-if に段階的プログレス表示を追加。briefing系にサマリービジュアルを追加。デモシナリオを感情の山を意識した順序で固定する。

### テーマF: 価格・ROI論理の脆弱性
- 関連問題: P1-5, Logical P0-2（価格論理）
- 根本原因: 価格は競合比較から逆算しているが、顧客が実際に支払うかの市場検証ゼロ。EC支払能力の上限（20万円）を超えた価格設定の内部矛盾を放置している。
- 対策方向: ROI試算シートを顧客提示可能な形式で作成し、計算前提を明示する。パイロット顧客でWTP（支払意欲）インタビューを実施して価格を検証する。

## Phase 3 への申し送り

**即日着手必須（デプロイ・デモ前に完了）:**
1. ec-what-if の離脱率データを `clip(0,1)` で修正し再学習。R² ≥ 0.3 を達成してから「キラー機能」を名乗る。
2. 全18URLを `curl` またはStreamlit Cloud管理画面でデプロイ状態確認。未デプロイURLはポータルから一時非表示化。
3. 「L3実装完了」の表現をビジネス資料全体で「L3プロトタイプ完成・パイロット検証前」に修正する。

**1週間以内（パイロット獲得準備）:**
4. 士業・EC各1社向けパイロット提案書（課題定義・提供機能・価格・期待ROI・契約期間）を作成。
5. 15分デモスクリプトを固定し（ec-what-if を中心に据えた感情の山を意識した順序）、週2回以上のデモ実施体制を構築。
6. ROI試算シートを顧客が実数値を入力できる形式で整備。

**技術負債への対処方針:**
- 新機能追加は一時停止。既存アプリの安定化（スモークテスト・エラーハンドリング統一・CI導入）を優先。
- コード重複（setup_japanese_font 共通化・EC共通マスター統合）は1〜2スプリントで対処。
- テスト導入（pytest）・GitHub Actions CI はパイロット顧客獲得後の第一Sprint目に組み込む。

**評価の健全化:**
- 品質スコアは採点基準を明文化するか、充足率（%）形式の客観指標に統一する。
- 「完了」の定義を「コードが存在する」ではなく「パイロット顧客1社での実データ運用が6週間継続した」に引き上げる。
- パイロット顧客獲得から計測を開始し、月次レビューの実績（顧客が具体的なアクションを取った回数）をL3到達の真の指標とする。
