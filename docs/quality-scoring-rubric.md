# MVP品質スコア採点ルーブリック

> 作成日: 2026-04-08
> 対象: L3キラー機能MVP（士業3本・EC3本）
> 目的: 「品質スコア91-95/100」の根拠を明示し、第三者再現性を確保

---

## 1. 採点が不透明だった問題

これまで本プロジェクトでは「品質スコア XX/100」と表現してきたが、以下の問題があった:
- 採点者: 主に開発エージェント自身（自己採点バイアス）
- 採点基準: 5軸×20点 = 100点と称していたが、各軸の評価基準が非公開
- 既存アプリ（充足率45-70%）と新規アプリ（91-95点）の乖離 → 採点軸が違う
- パイロット顧客0社の状態でビジネス訴求力20点満点付近 → 楽観バイアス

## 2. 新採点ルーブリック（6カテゴリ × 各20点満点）

### カテゴリA: 実装充足率（20点）
要件定義（synthesis参照）に対するコード実装の網羅性。

| 点数 | 基準 |
|:----:|------|
| 18-20 | 全機能実装 + エラーハンドリング |
| 14-17 | 主要機能実装、軽微な未実装あり |
| 10-13 | 主要機能の半数以上が動作するが、主要機能の一部が未実装 |
| 5-9 | プロトタイプ未満、骨格のみ |
| 0-4 | コードが動かない |

### カテゴリB: データ健全性（20点）
サンプルデータの妥当性、ML学習のR²、データ範囲の物理的妥当性。

| 点数 | 基準 |
|:----:|------|
| 18-20 | R² ≥ 0.5、データ範囲が物理的に妥当、欠損なし |
| 14-17 | R² 0.3-0.5、データ範囲が概ね妥当 |
| 10-13 | R² 0.0-0.3、データ範囲に一部問題 |
| 5-9 | R² < 0.0（ランダム以下）、または churn_rate > 1.0 等の論理破綻 |
| 0-4 | データ生成自体が失敗 |

### カテゴリC: UX完成度（20点）
Hero/CSS/タブ構成/サイドバー/レスポンシブの実装完成度。

採点項目（各2点）:
- [ ] Hero セクションが視認性高く実装
- [ ] KPIカード4枚以上
- [ ] タブ3つ以上で情報構造化
- [ ] サイドバーにフィルタ・説明・データ再生成ボタン
- [ ] 日本語フォント設定が機能
- [ ] CSS による色彩・余白統一
- [ ] エラー時の st.warning / st.error 利用
- [ ] CSV/Markdown ダウンロード機能
- [ ] グラフが2つ以上 (matplotlib)
- [ ] モバイル対応（st.columns 使用）

### カテゴリD: ドキュメント整合性（20点）
ビジネス資料・README・コミットメッセージ・コード内コメントの整合。

| 点数 | 基準 |
|:----:|------|
| 18-20 | 全ドキュメントとコードが整合、誇大表現なし |
| 14-17 | 軽微な不整合あり |
| 10-13 | 中程度の誇大表現あり（例: 「実装完了」に検証含意あり） |
| 5-9 | 重大な誇大表現（例: パイロット0社で「商業稼働」） |
| 0-4 | ドキュメント存在せず |

### カテゴリE: テスト・検証有無（20点）
スモークテスト、ユニットテスト、CI/CD、デプロイ実績の有無。

| 点数 | 基準 |
|:----:|------|
| 18-20 | スモーク + ユニット + CI + デプロイ実績あり |
| 14-17 | スモーク + デプロイ実績あり |
| 10-13 | スモークテストのみ（本プロジェクトの現在地） |
| 5-9 | ast.parse のみ |
| 0-4 | 検証一切なし |

### カテゴリF: デプロイ実績・顧客検証（20点）
Streamlit Cloud デプロイ + パイロット顧客検証の有無。

| 点数 | 基準 |
|:----:|------|
| 18-20 | 商業稼働中（売上実績あり） |
| 14-17 | パイロット顧客1社以上での運用検証完了 |
| 10-13 | デプロイ済み + 内部検証のみ |
| 5-9 | デプロイ済みだが未検証 |
| 0-4 | 未デプロイ（本プロジェクトの現在地） |

## 4. 旧スコア vs 新スコアの差異

| アプリ | 旧スコア | 新スコア | 主な減点理由 |
|--------|:-------:|:--------:|------------|
| shigyou-office-dashboard | 91/100 | 82/120 (68%) | デプロイ実績0、CI不在 |
| shigyou-ltv | 92/100 | 80/120 (67%) | データR²未公開、デプロイ0 |
| shigyou-briefing | 91/100 | 80/120 (67%) | デプロイ0 |
| ec-executive-dashboard | 95/100 | 83/120 (69%) | デプロイ0、テスト不足 |
| ec-what-if | 94/100 | 78/120 (65%) | データ欠陥履歴あり、デプロイ0 |
| ec-monthly-briefing | 95/100 | 80/120 (67%) | デプロイ0 |

平均: 旧 93/100 → 新 81/120 (67%)

## 5. 解釈

「91-95/100」というスコアは「実装の局所完成度」のみを反映していた。新採点では:
- 実装完成度: 引き続き高評価（17-18/20）
- データ健全性: ec-what-if が当初不合格、修正後も警告水準
- デプロイ実績: 全アプリ4/20（未デプロイ）
- 顧客検証: 全アプリ未実施

これは「プロトタイプ品質は高い、商業価値は未検証」という実態を正しく表現する。

## 6. 改善目標

カテゴリF（デプロイ・顧客検証）を 4 → 14 へ引き上げる施策:
1. パイロット提案書テンプレート作成（P1-F で実施）
2. 15分デモスクリプト作成（P1-G で実施）
3. パイロット顧客1社獲得（次フェーズ）
4. 6週間の運用検証（次フェーズ）

これにより全6アプリが 80/120 → 90+/120 へ到達可能。
