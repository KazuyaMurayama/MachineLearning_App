## Part 4: クロスチェック体制

### 4.1 チェック方法の設計

§8の数値を3つの独立した方法で検証する。

| チェック方法 | 目的 | 検証対象 |
|------------|------|---------|
| **① 時間予算積み上げ** | 全活動が月32hに収まるか | 各Phase×活動項目の合計 |
| **② 売上整合性検証** | MRR = 顧客数 × ARPU、二重計上なし | 月次テーブルの数学的整合 |
| **③ チャーン数学検証** | 解約が正しく織り込まれているか | 純増 = 新規 - (既存×チャーン率) |

| **⑤ 感度分析** | 前提が崩れた場合のインパクト | 4つの主要変数を変動させる |

### 4.7 チェック⑥ 第2クロスチェック（より厳格な検証）

第2の独立検証エージェントによるストレステスト結果。§9の数値に対し、さらに厳しい視点で検証。

#### 指摘1: ARPU ¥80Kは楽観的

初期顧客はLightプラン（¥50K）から始まる傾向が強い。M12時点のARPU ¥80Kを達成するには、7社中4社以上がStandard以上である必要があるが、現実的には**ARPU ¥70K程度**が妥当。

| 時点 | §9の想定ARPU | 修正ARPU | MRRへの影響 |
|------|-------------|---------|-----------|
| M12 | ¥80K | **¥70K** | ¥560K→¥490K（-12.5%） |
| M24 | ¥93K | **¥85K** | ¥1,120K→¥1,020K（-8.9%） |

#### 指摘2: 顧客上限は12社→10社

サポート時間は顧客数に対して**非線形**に増加する（問い合わせの連鎖、機能要望の調整等）。12社時のサポート時間を再計算:

```
§9の想定:  12社 × 0.8h = 9.6h
現実的推定: 12社 × 1.2h = 14.4h（非線形効果+0.4h/社）
→ 残り活動時間: 32h - 14.4h = 17.6h → コンサル+営業+開発で限界

10社に制限した場合:
  10社 × 1.0h = 10.0h
  残り: 32h - 10h = 22h → 十分な余裕
```

**修正:** 安全な顧客キャップは**10社**。11社目以降は自動化が十分に機能してから。

#### 指摘3: 経費の過少計上

§9の経費（年間¥40-85万）には以下が未計上の可能性:

| 費目 | 月額 | 年額 |

| **追加経費合計** | **¥5-15万** | **¥60-180万** |

**修正:** 24ヶ月の経費を¥130万→**¥250-310万**に上方修正。利益への影響: -¥120-180万。

### 追加5: リスク対応のフェーズ別優先順位（時間制約への解）

月32hで7リスク全てに同時対応は不可能。フェーズごとに集中すべきリスクを明確にする:

```
  → 契約書とプロセスで事前仕組み化。発生時のみ対応
```
