## 3.2 キラー機能: 逆SHAPエンジン

### コンセプト

```
通常SHAP:  予測結果 → 要因分析（Why?）
逆SHAP:    目標設定 → 改善アクション提案（How?）
```

**これが「これなしでは仕事できない」機能になる理由:**
- 予測だけなら他ツールでもできる
- 「具体的に何をすべきか」まで教えてくれるツールは**存在しない**
- 経営判断に直結するアクショナブルな情報

### 業界別の逆SHAP活用例

| 業界 | 質問 | 逆SHAPの回答 |
|------|------|-------------|
| 製造業 | 見積を10%下げるには？ | 材質変更で¥180K削減、ロット増で¥120K削減 |
| 物流業 | 来月の受注を安定化するには？ | 大口顧客Aへの事前確認が最も効果的 |
| 不動産 | 空室を1ヶ月短縮するには？ | Wi-Fi設置(¥50K)がROI最大 |

## 3.4 技術的差別化

### ポジショニングマップ

```
              説明可能性 高い
                   │
     当アプリ ★    │    ● 統計ソフト（SPSS等）
                   │
                   │
─────────────────┼─────────── 操作の簡単さ
    ● DataRobot    │    ● ChatGPT
                   │
    ● AWS          │
     SageMaker     │
              説明可能性 低い
```

### LightGBM + SHAP の3層差別化

| 層 | 内容 | 競合優位性 |
|----|------|-----------|
| **精度層** | LightGBM = 構造化データ最強。Kaggle上位の60%以上がGBDTベース | 少量データ（中堅企業に最適）に特に強い |
| **説明層** | TreeSHAPで厳密なShapley値を高速計算 | 個別予測+グローバル重要度の両方を提供 |
| **アクション層（新）** | 逆SHAP + What-Ifシミュレーション | **唯一無二。競合にはない** |

### データ品質自動診断機能

**ユーザー離脱の最大原因: データ品質が悪く精度が出ない → 失望 → 離脱**

```
┌──────────────────────────────────────────┐
│ 📊 データ品質レポート                     │
│                                           │
│ 総合スコア: 72/100  🟡 良好（改善推奨）   │
│                                           │
│ 📋 検出された課題:                        │
│ ⚠️ '材料コード'の欠損率が35% → [自動補完]│
│ ⚠️ '単価'に23件の外れ値 → [自動クリップ] │
│ ℹ️ '部品名'のカテゴリが250種 → [グループ化]│
│                                           │
│ [すべて自動修正] [個別に確認] [無視して続行]│
└──────────────────────────────────────────┘
```

**スコアリング基準:**
- 完全性（欠損率）: 30%
- 一貫性（型の統一性）: 20%
- 正確性（外れ値率）: 20%
- 適時性（データ鮮度）: 15%
- 十分性（行数・列数）: 15%

## 3.5 プロダクト進化ロードマップ

### Phase 1: MVP → PMF（0-6ヶ月）

**目標:** 製造業で30社の有料顧客を獲得し、PMFを確認

| 機能 | 優先度 | 実装工数 | 価値 |
|------|--------|---------|------|
| CSVアップロード + 自動学習 | P0 | 完了 | ★★★★★ |
| SHAP分析 | P0 | 完了 | ★★★★★ |
| データ品質自動診断 | P0 | 2週間 | ★★★★★ |
| 業界別サンプルデータ | P0 | 1週間 | ★★★★ |
| 逆SHAP提案エンジン（基本版） | P1 | 3週間 | ★★★★★ |
| 予測信頼区間 | P1 | 1週間 | ★★★★ |
| PDF/PPTレポート生成 | P1 | 2週間 | ★★★★★ |

**PMF判定基準（品質チェック反映: 創業者効果を排除）:**
- 週次アクティブ率 > 40%
- NPS > 40
- **セルフサービス獲得顧客**のリテンション率 > 85%
- 無料→有料転換率 > 10%

### Phase 2: Growth（6-18ヶ月）

**目標:** 3業界で500社、ARR 3億円

**主要機能:**
1. **業界ベンチマーク** — 匿名化データの蓄積、他社比較
2. **API連携** — 基幹システムからの自動データ取得
3. **チーム機能** — 複数ユーザー、権限管理
4. **自動再学習** — 新データ追加時にモデル自動更新
5. **Streamlit → Next.js + FastAPI移行**

**データフライホイール:**
```
ユーザー増加 → データ蓄積（匿名化）→ ベンチマーク精度向上
    ↑                                         ↓
新規ユーザーの魅力増大 ←── ベンチマーク比較の価値向上
```

### Phase 3: Moat構築（18-36ヶ月）

**目標:** 競合が真似できない堀の構築

```
Layer 4: エコシステムMoat（パートナー網、API連携、業界標準化）
Layer 3: データMoat（業界ベンチマーク、匿名化集合知）
Layer 2: ナレッジMoat（業界テンプレート、改善パターン）
Layer 1: テクノロジーMoat（逆SHAP、ドメイン特化AutoML）
```

**分類モデル追加: Phase 2後半（12-18ヶ月目）**
- 判断基準: 有料顧客200社超 + 顧客要望30%以上
- TAM拡大効果: 1.5-2倍（+約794億円）

## 3.7 追補: プロダクトエージェント（詳細版）からの実装仕様

> コードベースを精査した詳細版エージェントが、具体的な実装仕様を提案。

### 正直な差別化評価

**LightGBM + SHAP の組み合わせ自体は差別化にならない。** DataRobotもAutoMLも同じことができる。

真の差別化は以下の3点に再定義すべき：

| 差別化ポイント | 内容 | 競合の弱点 |
|-------------|------|-----------|
| 日本の中堅企業特化 | 日本語CSV即投入、商習慣対応（年度・祝日・消費税） | AutoML/DataRobotは日本語対応が弱い |
| 説明可能性ファースト | LightGBM固定で常にSHAP説明可能 | AutoMLは精度優先でブラックボックスモデルを出力しがち |
| オンプレ対応 | Streamlitアプリはシングルファイルでデプロイ可能 | クラウドAutoMLは自社サーバーで動かせない |

### 具体的な追加機能仕様

#### 見積シミュレーター（フォーム入力 → リアルタイム予測）

```python
# 学習データのカラムからUIフォームを自動生成
# 数値型 → st.number_input() or st.slider()
# カテゴリ型 → st.selectbox()（ユニーク値をオプションに）
# 入力変更ごとにリアルタイム予測値更新
# SHAP forceplotで「なぜこの価格か」を1件ごとに説明
```

CSV不要で手軽な予測体験を提供。営業現場での見積確認にも使える。

#### 時系列特徴量の自動生成（物流ユースケース必須）

```python
# 日付カラム自動検出 → 以下を自動生成：
# year, month, day, dayofweek, is_weekend, quarter
# 祝日フラグ（jpholidayライブラリ連携）
# ラグ特徴量（lag_1, lag_7, lag_30）
# 移動平均（rolling_7, rolling_30）
```

現在のアプリには時系列特有の特徴量エンジニアリングが一切ない。物流ユースケースで致命的。

#### 日本語Excel特有のデータクリーニング

```python
def clean_numeric_strings(df):
    """日本語Excel特有の数値文字列をクリーニング"""
    # "1,234" → 1234
    # "￥5,000" → 5000
    # "１２３" → 123（全角→半角変換）
```

日本のExcelデータでは頻出パターン。対応しないと精度が出ない。

#### 分類モデル自動判定

```python
def detect_task_type(y):
    """目的変数から回帰/分類を自動判定"""
    unique_ratio = y.nunique() / len(y)
    if y.dtype == 'object' or unique_ratio < 0.05:
        return 'classification'
    return 'regression'
```

Phase 2初期（4-5ヶ月目）に追加推奨。LightGBMClassifierは既にライブラリに含まれているため実装コストが低い。

### 「ストーリーフロー」UIへの転換

現状のタブ並列UIを、ユーザーの業務フローに合わせたステップ直列フローに変更：

```
現状: タブ並列（データ | 評価 | SHAP | 予測）
  ↓
改善: ステップ直列
  Step 1: データ確認（品質スコア表示）
  Step 2: 学習＆評価（自然言語サマリー）
  Step 3: 結果活用（3サブモード）
    3a. バッチ予測（CSV一括）
    3b. 個別予測（フォーム入力）
    3c. 分析レポート（PDF出力）
```

### AHA Momentの再設計

現状のR²表示は「だから何？」となる。以下に変更：

1. **「3クリック60秒で予測完了」** — 目的変数を自動推定して手順を削減
2. **「SHAPの意外な発見」ハイライト** — 上位3特徴量をカード形式で自然言語説明
3. **「Excelとの精度比較」** — ユーザーの手動予測との比較表示（「AI予測はあなたの予測より23%正確です」）
